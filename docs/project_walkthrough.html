    Part 3
    The Data — 1,025,275 Shipments
    We can't use Amazon's real data (confidential). Instead, we generated realistic synthetic data — every number calibrated to public industry benchmarks from FreightWaves, Cass, ATRI, and BTS.

    Why Not Real Data?

    Amazon's freight data is proprietary and under NDA. No public shipment-level dataset exists with carrier identity, cost breakdowns, and defect flags at this granularity. Synthetic data lets us control exactly what the data looks like while demonstrating the same analytical skills.

    Why It's Still Credible

    Every parameter in src/config.py has a citation comment. OTP baseline = 87% → FreightWaves SONAR. Cost = $2.43/mile → Cass Freight Index 2024. An interviewer can challenge any number and you point to the source.

    -- Every row in fact_shipment has these fields: SELECT shipment_id, -- Unique ID: SHP-20251201-0000001 carrier_id, -- Which trucking company (52 carriers) lane_id, -- Origin→Destination pair (225 lanes) shipper_id, -- Which seller shipped it (340 shippers) pickup_appt_ts, -- Scheduled pickup time pickup_actual_ts, -- Actual pickup (NULL 0.4% → tracking gap) on_time_pickup, -- TRUE if actual ≤ appt + 15 min tolerance dwell_minutes, -- Time at facility (can be negative = early) defect_flag, -- TRUE if quality issue occurred trailer_utilization_pct, -- 0.0–1.0, how full the trailer was linehaul_cost_usd, -- Base transport cost (DECIMAL, not float!) fuel_surcharge_usd, -- ~20% of linehaul accessorial_cost_usd, -- Detention, rescheduling, etc. total_cost_usd -- EXACT: linehaul + fuel + accessorial = total FROM fact_shipment LIMIT 1;
    📉

    Power-Law Shippers

    Top 20 shippers = 54.3% of all volume. Just like real retail: a few giant sellers dominate. Modeled with Pareto distribution (alpha=1.5).

    🎯

    4 Carrier Tiers

    Strategic (best OTP, cheapest), Core, Tactical, Spot (worst OTP, most expensive). Each tier has different OTP baselines, defect rates, and cost modifiers.

    🕳️

    Missing Data

    0.41% NULL pickup timestamps (EDI/API failures), 0.10% negative dwell (early arrivals). Real data is always messy. Our queries handle both gracefully.

    📅

    Q4 Seasonal Lift

    Oct–Dec shows +12% cost-per-mile (confirmed statistically in q33). The SARIMAX forecast captures this pattern automatically.

    💰

    Exact Cost Identity

    linehaul + fuel + accessorial = total. Exactly. Across all 1,025,275 rows. Using Python Decimal (not float). Zero tolerance — proven by a test.

    📉

    Churn Signals

    5 top-20 shippers show >30% volume decline in last 90 days. The Shipper Health Score detects these before churn becomes irreversible.

    🚨 The Injected Anomaly — Our "Hidden Bug"

    We deliberately planted a 14-day period where one Spot carrier on 3 specific lanes drops to 60% OTP (network average: 85.5%). The anomaly detector MUST catch this — and it does (z-score = -3.1, flagged within 3 days).

    -- From src/config.py — the planted anomaly: INJECTED_ANOMALY = { "carrier_tier": "Spot", # Worst-performing tier "lane_indices": [0, 1, 2], # 3 specific lanes "month_offset": 7, # Month 8 (Dec 2025) "duration_days": 14, # 2 weeks "degraded_otp": 0.60 # Drops to 60% OTP }

    Interview story: "I planted a known anomaly and verified my detector caught it — that's how I validated the system."

    Part 5
    35 SQL Queries — The Analytical Engine
    35 numbered, business-question-first SQL queries. Click a category to see examples. These prove SQL mastery from basic aggregations to statistical hypothesis testing.
    -- q07: Weekly OTP with rolling 4-week average (Window Function) WITH weekly AS ( SELECT d.year, d.week_of_year, MIN(d.date) AS week_start, AVG(CASE WHEN f.on_time_pickup THEN 1.0 ELSE 0.0 END) AS otp_rate FROM fact_shipment f JOIN dim_date d ON f.shipment_date_key = d.date_key GROUP BY d.year, d.week_of_year ) SELECT year, week_start, ROUND(otp_rate * 100, 1) AS otp_pct, -- Rolling 4-week average using window function: ROUND(AVG(otp_rate) OVER ( ORDER BY year, week_of_year ROWS BETWEEN 3 PRECEDING AND CURRENT ROW ) * 100, 1) AS rolling_4w_avg_pct, 90.0 AS network_target_pct FROM weekly ORDER BY year, week_of_year;

    💡 Technique: CTE + Window function with ROWS BETWEEN frame. Used to show trend deterioration while smoothing weekly noise.

    -- q14: Lane Profitability Index vs $2.43/mile benchmark (Cass 2024) SELECT l.lane_id, l.origin_city || ' → ' || l.dest_fc_id AS lane, l.lane_type, COUNT(*) AS shipment_count, ROUND(AVG(f.total_cost_usd / NULLIF(f.distance_miles, 0)), 2) AS observed_cpm, 2.43 AS benchmark_cpm, -- Profitability index: positive = below benchmark (good) ROUND((2.43 - AVG(f.total_cost_usd / NULLIF(f.distance_miles,0))) / 2.43 * 100, 1) AS profitability_index_pct, CASE WHEN AVG(f.total_cost_usd/NULLIF(f.distance_miles,0)) > 2.43 THEN 'Above benchmark' ELSE 'Below benchmark' END AS cost_status FROM fact_shipment f JOIN dim_lane l ON f.lane_id = l.lane_id GROUP BY l.lane_id, l.origin_city, l.dest_fc_id, l.lane_type, l.distance_miles HAVING COUNT(*) >= 100 ORDER BY profitability_index_pct ASC;
    -- q21: LTL shipments that look like FTL (modal mismatch) -- Business question: Are we paying LTL rates for FTL-sized loads? SELECT l.origin_city || ' → ' || l.dest_fc_id AS lane, COUNT(*) AS ltl_shipment_count, ROUND(AVG(f.trailer_utilization_pct) * 100, 1) AS avg_utilization_pct, ROUND(AVG(f.weight_lbs), 0) AS avg_weight_lbs, 'Consider FTL conversion' AS recommendation FROM fact_shipment f JOIN dim_lane l ON f.lane_id = l.lane_id JOIN dim_service_type st ON f.service_type_id = st.service_type_id WHERE st.service_name = 'LTL' GROUP BY l.lane_id, l.origin_city, l.dest_fc_id HAVING AVG(f.trailer_utilization_pct) > 0.70 -- 70%+ fill = FTL territory AND AVG(f.weight_lbs) > 30000 -- Heavy loads = FTL candidate AND COUNT(*) >= 20 ORDER BY avg_utilization_pct DESC;

    🎓 These 3 queries are the crown jewels. Any analyst can compute an average. Running a statistical test inside SQL and interpreting it in business language is what separates a Business Analyst from a reporting analyst.

    -- q31: Paired t-test — Is Strategic vs Tactical OTP difference real? -- H0: No difference. If |t| > 1.96, reject at alpha=0.05. WITH lane_tier_otp AS ( SELECT f.lane_id, c.carrier_tier, AVG(CASE WHEN f.on_time_pickup THEN 1.0 ELSE 0.0 END) AS lane_otp FROM fact_shipment f JOIN dim_carrier c ON f.carrier_id = c.carrier_id WHERE c.carrier_tier IN ('Strategic', 'Tactical') GROUP BY f.lane_id, c.carrier_tier HAVING COUNT(*) >= 30 ), paired AS ( SELECT s.lane_otp - t.lane_otp AS diff FROM lane_tier_otp s JOIN lane_tier_otp t ON s.lane_id = t.lane_id WHERE s.carrier_tier = 'Strategic' AND t.carrier_tier = 'Tactical' ), stats AS ( SELECT COUNT(*) AS n, AVG(diff) AS mean_diff, STDDEV(diff) AS std_diff FROM paired ) SELECT ROUND(mean_diff * 100, 2) AS mean_otp_diff_pp, ROUND(mean_diff / (std_diff / SQRT(n)), 3) AS t_statistic, CASE WHEN ABS(mean_diff / (std_diff/SQRT(n))) > 1.96 THEN 'REJECT H0 — Significant difference (p<0.05)' ELSE 'FAIL TO REJECT H0' END AS conclusion FROM stats; -- Result: t-statistic >> 1.96 → Strategic IS significantly better ✓

    🔍 Z-Score Anomaly Detection

    For OTP % (bounded 0–100). Computes an 8-week rolling mean and standard deviation per lane. Flags weeks where |z| > 2.5 (~top 1.2% most extreme).

    z = (observed_otp - rolling_mean) / rolling_std
    if |z| > 2.5 → flag as anomaly
    Result: 659 anomalies flagged
    Severity Distribution
    Critical (|z|>3.5)
    High (|z|>3.0)
    Medium (|z|>2.5)

    📦 IQR Anomaly Detection

    For Cost-per-Mile (heavy right tail). Z-score doesn't work for heavily skewed data. IQR (Interquartile Range) method uses fences instead — robust to outliers.

    Q1, Q3 = 25th/75th percentile of CPM
    IQR = Q3 - Q1 (middle 50% range)
    upper_fence = Q3 + 1.5 × IQR
    Result: 176 anomalies flagged

    Why two methods? Because no single method works for all data shapes. OTP is bounded and roughly normal → z-score. CPM has a long tail → IQR. Using both shows statistical sophistication.

    📈 SARIMAX Forecasting

    Seasonal ARIMA for OTP and Cost. Named so because it handles: Seasonal, Auto-Regressive, Integrated, Moving Average patterns.

    Part 10
    Interview Cheat Sheet
    Word-for-word answers for the four most likely questions.

    ❝ Tell me about a project where you used data to drive a decision. ❞

    "I built an inbound first-mile analytics platform analyzing over a million shipment records across 225 lanes, 52 carriers, and 340 shippers. I discovered that Monday pickups run 5 percentage points below mid-week OTP — not because of volume spikes, but because of weekend backlog creating morning congestion. I validated this with day-of-week SQL analysis (q06) and a rolling trend view (q07), then recommended staggered pickup windows. I also found that Spot-tier carriers on long-haul lanes produce 80% more defects than the network average, confirmed by a chi-square test at p<0.05 (q32), and recommended restricting them from those lanes. The project runs end-to-end from a single command — link is in my application materials."

    ❝ Walk me through your SQL skills with an example. ❞

    "Let me show you three queries at increasing complexity. First, q07 uses a CTE to compute weekly OTP, then a window function with ROWS BETWEEN 3 PRECEDING AND CURRENT ROW for a rolling 4-week average — you see the trend before it becomes a crisis. Second, q14 computes a lane profitability index by comparing observed CPM to the Cass 2024 benchmark of $2.43/mile, using NULLIF to handle zero-distance edge cases and HAVING to filter low-volume lanes. Third — and this is the one I'm most proud of — q31 implements a paired t-test entirely in SQL, no Python. It pairs Strategic and Tactical carriers by lane, computes the mean OTP difference, standard deviation, and t-statistic, then prints 'REJECT H0' or 'FAIL TO REJECT H0' in the result set. The t-statistic comes out way above 1.96, confirming that Strategic carriers really are significantly better — it's not just random noise."

    ❝ Tell me about a hard technical bug you encountered. ❞

    "My cost identity check — linehaul + fuel + accessorial must exactly equal total_cost — was failing by pennies. Something like $135.20 stored as $135.19999999999998. The root cause was Python float arithmetic accumulating rounding error. $100.10 + $20.03 + $15.07 in IEEE 754 floats sometimes gives you 135.19999... not 135.20. I fixed it by switching the entire cost computation in generate_data.py to Python's Decimal type with 2-decimal-place quantization (quantize(Decimal('0.01'))). The identity now holds exactly — to the penny — across all 1,025,275 rows, proven by test_warehouse_integrity.py::test_cost_identity_exact which asserts zero rows with a mismatch. Lesson: never use floats for money."

    ❝ How do you handle messy or incomplete data? ❞

    "I built messiness into the data intentionally so I'd have to handle it properly. 0.41% of pickup timestamps are NULL — tracking failures where the EDI feed dropped. My OTP query uses on_time_pickup BOOLEAN (pre-computed during load) and my anomaly detector only uses non-NULL records when computing rolling averages. 0.10% of dwell times are negative — drivers who arrived before their appointment window. My KPI queries use FILTER (WHERE dwell_minutes > 0) when computing averages, and q35 is a dedicated negative dwell audit query. The quality_checks.py module validates all of this on every pipeline run — it asserts that NULL pickups stay below 0.5% and negative dwell below 0.15%, so if data quality degrades I catch it before it silently corrupts downstream analytics."
